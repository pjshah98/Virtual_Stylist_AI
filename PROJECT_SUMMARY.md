# Virtual Stylist Web App — Technical Project Summary

This document gives a concise technical overview of the Virtual Stylist web application: architecture, data pipeline, and key integration points for further development and maintenance.

---

## 1. Project Overview

**Virtual Stylist** is a Flask web app that suggests outfit combinations based on user preferences (top wear, bottom wear, shoes, and preferred colors). It uses a pre-trained scikit-learn Random Forest model and a simple recommendation pipeline. The UI is a two-screen flow: a preferences form and a recommendations results page.

- **Purpose**: Outfit recommendations from user-selected categories and colors.
- **Stack**: Python 3, Flask, Flask-WTF, Jinja2, HTML/CSS/JS, pandas, scikit-learn, pickle for model persistence.
- **Run**: `python run.py` (Flask dev server, debug mode).

## 3. Data and ML Pipeline

**Dataset (`ml/dataset/fashion_data_large.csv`)**

- **Columns**: `top_wear`, `bottom_wear`, `shoes`, `color`, `label`
- **color**: Multi-label, comma-separated (e.g. `"Blue, Green, Black"`).
- **label**: Binary (0/1), used as target in the notebook.

**Training (in `ml/fashion_trends.ipynb`)**

- Read CSV; expand `color` into one-hot columns (e.g. Red, Blue, Black, …).
- Categorical columns (top_wear, bottom_wear, shoes) are one-hot encoded via `pd.get_dummies`.
- `feature_names = features.columns.tolist()` is saved with the model so inference uses the same columns.
- Train/test split (80/20), GridSearchCV over RandomForestClassifier (n_estimators, max_depth, min_samples_split, min_samples_leaf), cv=3.
- Save to `ml/models/recommendation_model.pkl`: `(best_model, feature_names)`.

**Inference (in `app/utils.py`)**

- Load model and `feature_names` via `app/models.load_model()`.
- Build one row: user’s `top_wear`, `bottom_wear`, `shoes` plus binary color columns (1 if color in user’s selected list, else 0).
- DataFrame is one-hot encoded and aligned to `feature_names` (reindex, fill_value=0).
- `model.predict_proba(input_df)[0]` is computed but **not currently used** to choose specific items (see below).
- **Important**: The five “recommendations” are **generated by random choice** from fixed lists of top_wear, bottom_wear, shoes, and the user’s selected colors. So the ML output influences the pipeline (e.g. feature space) but the final outfit strings are randomized, not directly from the model’s prediction. Improving “smart” recommendations would mean using the model’s scores or predictions to rank/select items.

## 4. Backend Behavior

**`app/__init__.py`**

- Flask app factory `create_app()`: sets `SECRET_KEY`, registers blueprint `main`.

**`app/routes.py`**

- Single blueprint `main`.
- `GET /` or `POST /`: show `PreferencesForm`. On valid submit, build `preferences` dict (top_wear, bottom_wear, shoes, color list), call `get_recommendations(preferences)`, render `recommendations.html` with `recommendations` list. Otherwise render `index.html` with `form`.

**`app/forms.py`**

- **PreferencesForm**:  
  - `top_wear`: SelectField (Shirt, T-shirt, Blouse, Hoodie, Sweater, Jacket, Tank top, Polo, Cardigan, Vest).  
  - `bottom_wear`: SelectField (Jeans, Shorts, Skirt, Trousers, Leggings, Sweatpants, Chinos, Capris, Joggers, Culottes).  
  - `shoes`: SelectField (Sneakers, Boots, Flats, Heels, Sandals, Loafers, Moccasins, Derby, Oxford, Brogue).  
  - `color`: MultiCheckboxField (Red, Blue, Black, Green, Yellow, Pink, White, Gray, Purple, Orange).  
  - `submit`: SubmitField.
- All single-choice fields have `DataRequired()`; color is optional in the form but the frontend (scripts.js) requires at least one color.

**`app/utils.py` — `get_recommendations(preferences)`**

- Loads model and feature_names.
- Builds `input_features`: top_wear, bottom_wear, shoes + one binary per color.
- Converts to DataFrame, one-hot encodes, reindexes to `feature_names`, fill_value=0.
- Runs `model.predict_proba(input_df)[0]`.
- Generates **5** recommendations: for each, randomly picks top, bottom, shoes from hardcoded lists (top_wear_options, bottom_wear_options, shoes_options) and three colors from `selected_colors` (user’s list or all colors). Tries up to 50 attempts to get three distinct colors when the user selected ≥3. Each recommendation is a string like: `"You can wear a {color} {top} with {color} {bottom} and {color} {shoes}."`
- Returns list of 5 strings.

**`app/models.py`**

- `load_model()`: opens `../ml/models/recommendation_model.pkl` (path relative to `app/`), returns `(model, feature_names)`.

## 5. Frontend

**Templates**

## 8. Notable Gaps and Improvement Hooks

- **Recommendations are random**: Model’s `predict_proba` is not used to select or rank outfit pieces; recommendations are random from fixed lists. Any “smarter” feature (e.g. ranking by model score, filtering by compatibility) would need changes in `app/utils.py` and possibly the training pipeline/labels.
- **Feature alignment**: Inference must match training: same one-hot columns and order as `feature_names`. Adding/removing categories in the form or dataset requires retraining and saving updated `feature_names`.
- **ml/recommendation_engine.py**: Empty; all logic is in `app/utils.py`. Could be refactored into `ml/recommendation_engine.py` and imported by the app.
- **Form choices vs dataset**: Form has more options (e.g. shoes: Flats, Heels, Moccasins, Oxford, Brogue) than the hardcoded lists in `utils.py` (e.g. shoes_options: Sneakers, Boots, Sandals, Loafers, Derby). Inconsistency can confuse users.
- **No images**: Recommendations are text-only; no product images or thumbnails.
- **Single route**: No API layer; no auth; no history or saved preferences.
- **Tests**: Only one route test; no tests for `get_recommendations` or model loading.

## 9. Quick Reference for Common Tasks

- **Add a new clothing option**: Add to form choices in `forms.py`, add to the corresponding options list in `utils.py`, and if the ML model should use it, add to the dataset and retrain in the notebook, then save a new pkl with updated feature_names.
- **Change number of recommendations**: Change `top_n` in `app/utils.py` (e.g. from 5 to 10).
- **Styling**: Edit `app/static/css/styles.css` and templates; base layout in `app/templates/base.html`.
- **Form validation**: Server-side in `forms.py` (validators); client-side in `app/static/js/scripts.js`.
- **Retrain model**: Run all cells in `ml/fashion_trends.ipynb` (from `ml/` so paths to `dataset/` and `models/` work); ensure `recommendation_model.pkl` and feature_names stay in sync with the form and `utils.py` feature building.
